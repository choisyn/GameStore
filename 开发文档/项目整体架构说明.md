# 项目整体架构说明

> 文档用途：集中说明本项目的框架、结构、参数、流程、技术选型与设计思路，作为开发说明、论文写作和答辩陈述的统一参考。
>
> 编写依据：`pom.xml`、`src/main/resources/application.properties`、`src/main/java/com/gamestore/**`、`src/main/resources/templates/**`、`src/main/resources/static/**`、`开发文档/SQL脚本/**` 以及本地开发库 `bishe` 的当前数据。
>
> 统计时间：2026-03-18

## 1. 项目定位

本项目名称为“游戏社区与数字商品商城系统”，是一个面向毕业设计场景的小型 Web 单体应用。系统将游戏商城、讨论广场、社区板块、游戏攻略、积分成长、后台管理等功能放在同一个 Spring Boot 工程内实现，目标是完成一个可本地部署、可演示、具备完整业务闭环的社区型游戏平台原型。

项目不是企业级分布式系统，当前设计标准以“小型课程/毕业设计系统”为准，主要特点如下：

- 部署方式为单体应用，前后端同仓库、同进程运行。
- 数据存储使用单库 MySQL，不拆分微服务，不使用消息队列。
- 认证方式采用自定义 Session Token，不依赖完整 Spring Security 登录体系。
- 推荐、成长、签到、积分商城等创新功能以规则计算和行为日志为主，强调可解释性和可演示性。
- 适用规模以本地教学演示和中小样例数据为主，不以高并发、高可用、多租户为目标。

### 2.4 已保留但当前未作为主线启用的配置

项目中仍保留了部分 Redis、JWT、Spring Security 相关依赖痕迹或配置项，但当前主线业务并不依赖它们完成登录与授权。现阶段真实生效的是：

- 自定义 Session Token 登录态
- Cookie / Header 双通道取 Token
- 基于 Controller + Service 的业务权限判断

因此在项目说明或论文中，不应将本项目描述为“Redis 会话中心”“JWT 鉴权系统”或“完整 Spring Security 安全体系”。

## 3. 整体架构

系统采用典型的单体分层架构：

```text
浏览器
  -> Thymeleaf 页面 / 静态 HTML / JavaScript
  -> Controller 接收请求
  -> Service 处理业务逻辑
  -> Repository 访问数据库
  -> MySQL 持久化
  -> 统一响应对象返回前端
```

### 3.1 分层说明

- `controller`
  - 负责页面路由、REST API、权限入口判断、参数接收与响应返回。
- `service`
  - 负责业务逻辑实现，如登录、购物车、结算、推荐、发帖、积分、攻略、打赏等。
- `repository`
  - 负责数据库查询，基于 Spring Data JPA。
- `entity`
  - 负责实体映射，对应数据库表结构。
- `dto`
  - 负责请求参数和响应结果封装。
- `exception`
  - 负责业务异常与全局异常处理。
- `util`
  - 包含统一响应工具、游戏名称格式化等通用工具。
- `config`
  - 负责静态资源映射等基础配置。

### 3.2 架构特点

- 页面和 API 并存：既有模板页面，也有供前端脚本调用的 JSON 接口。
- 商城与社区共库：用户、游戏、订单、帖子、评论、积分等共用同一数据库。
- 模块组合式扩展：推荐、成长、签到、积分商城、内容打赏是基于现有用户与内容数据叠加出来的扩展层。
- 历史兼容修复：`LegacySchemaRepairRunner` 会在启动时自动修补旧表字段、索引与外键，降低历史脚本迁移成本。

这说明项目已经具备完整主线，但在前端页面组织上仍然保留了“边开发边迭代”的痕迹。

对应类：

- `AuthController`

需要注意的当前实现细节：

- 因此当前接口默认对前台开放的主要板块是 `讨论广场` 和 `游戏攻略`。

### 5.6 游戏攻略

对应类：

- `GameGuideController`
- `GuidePageController`
- `GameGuideService`
- `GameGuide`

- 攻略列表
- 攻略详情
- 精选攻略
- 攻略数量统计

发布规则：

- 必须绑定游戏
- 标题长度 5 到 200 个字符
- 正文至少 20 个字符
- 摘要最多 500 个字符
- 预计阅读时长默认 10 分钟，最大 600 分钟

### 5.7 创新功能：推荐、决策、成长、签到

对应类：

- `InnovationController`
- `InnovationService`
- `UserBehaviorLog`
- `UserBadge`

主要功能：

- 个性化推荐
- 购前决策分析
- 用户成长面板
- 每日签到
- 行为埋点记录
- 勋章发放

设计思路：

- 不引入复杂机器学习服务；
- 使用浏览、加购、购买、评论、发帖、签到等行为生成兴趣偏好；
- 再结合游戏标签、分类、价格、评分、社区热度，形成可解释推荐结果。

### 5.8 内容打赏

- 单次打赏积分必须大于 0

### 7.3 关键数据关系

- 一个用户可拥有多个会话、帖子、评论、订单、行为日志和勋章。
- 一个订单可包含多个订单项。
- 一个订单完成后可派生多个 `user_games` 入库记录。
- 一个游戏可通过 `game_categories` 关联多个分类。
- 一个帖子可对应多个评论，评论可通过 `parent_id` 形成回复结构。
- 一个社区板块可包含多个板块帖子与评论。
- 一个攻略可被多位用户打赏。

## 9. 核心业务流程

### 9.1 注册与登录流程

```text
用户填写注册信息
-> AuthController.register
-> AuthService 校验用户名/邮箱/确认密码
-> PasswordService 加密密码
-> 保存 users

用户登录
-> AuthController.login
-> AuthService 校验密码和状态
-> 删除旧 session
-> 生成新 token
-> 写入 user_sessions
-> 设置 SESSION_TOKEN Cookie
```

### 9.2 商城购买流程

```text
浏览游戏
-> 加入购物车 cart_items
-> 勾选结算项
-> 选择消费卡/积分抵扣
-> 生成 orders 与 order_items
-> 写入 user_games
-> 删除已结算购物车项
-> 增加 point_transactions
-> 返回订单结果
```

### 9.3 免费游戏领取流程

```text
访问免费游戏
-> 判断当前价格是否为 0
-> 校验未拥有
-> 生成 FREE 订单
-> 写入 order_items
-> 写入 user_games
-> 记录行为日志
```

### 9.4 讨论广场发帖流程

```text
用户登录
-> 前端可先上传图片
-> 图片保存到 uploads/forum/日期目录
-> 提交帖子标题、正文、图片 JSON、关联游戏
-> 保存 posts
-> 记录 CREATE_FORUM_POST 行为
```

### 9.5 推荐与成长流程

```text
用户浏览/加购/购买/发帖/评论/签到
-> 生成 user_behavior_logs
-> InnovationService 归纳标签与偏好得分
-> 结合游戏标签、分类、价格、评分、热度
-> 输出推荐列表、购前决策分析和成长面板
```

## 10. 设计思路与实现特点

### 10.1 单体架构优先

本项目选择单体架构而不是微服务，原因是：

- 毕业设计场景强调可实现、可运行、可演示；
- 单体结构更适合小规模开发和答辩演示；
- 可以显著降低部署复杂度、调试成本和文档负担；
- 便于在同一工程中完成商城、社区、攻略和后台的联动。

### 10.2 以真实可落地功能替代“企业级堆栈”

项目没有刻意引入缓存集群、分布式事务、消息中间件、搜索引擎等复杂设施，而是把精力放在：

- 商城闭环是否可跑通；
- 讨论与社区是否可互动；
- 用户行为是否可沉淀；
- 创新功能是否有可解释结果；
- 后台是否可以对主要数据进行管理。

这种取舍更适合毕业设计项目的完成度要求。

### 10.3 会话认证而非完整安全框架

当前系统采用自定义会话表 `user_sessions`，优点是实现直观、易于理解、方便排查；缺点是：

- 权限校验分散在各 Controller / Service 中；
- Cookie 安全性一般；
- 没有形成统一过滤器链和标准化安全配置。

因此它适合教学演示，但后续若继续演进，建议升级为正式安全方案。

### 10.4 多分类兼容设计

`games.category_id` 仍被保留，但系统同时引入 `game_categories` 多对多关系：

- 老数据与旧查询还能继续工作；
- 新功能可以支持一个游戏属于多个分类；
- 这体现了项目从早期版本向较完整版本平滑演进的设计思路。

### 10.5 行为日志驱动创新功能

推荐、成长、签到、勋章并非独立孤岛，而是围绕 `user_behavior_logs` 组织：

- 商城行为会影响推荐；
- 社区互动会影响成长任务与勋章；
- 签到会增加积分并参与活跃度判断；
- 打赏和积分兑换又进一步强化内容生态与商城联动。

这使得项目不仅是“商城 + 论坛”的简单拼接，而是形成了一个轻量级的互动闭环。

## 12. 当前限制与后续优化方向

### 12.1 现有限制

- 讨论广场页面与社区板块页面仍存在模板和静态页混用情况。
- 社区板块的部分前台页面模板尚未完全补齐。
- 登录态使用可被前端脚本读取的 Cookie，安全性一般。
- `show-sql=true` 更适合开发，不适合正式部署。
- 推荐逻辑以规则计算为主，尚未引入更细粒度的协同过滤或模型服务。
- 测试数量较少，缺少完整接口与事务级测试。

### 12.2 后续可扩展方向

- 统一前台页面组织方式，减少静态页与模板页混用。
- 引入更规范的安全配置，例如正式的 Spring Security 登录链路。
- 补充后台中的分类管理、轮播图管理与更多统计功能。
- 强化文件上传校验、异常提示和前端交互体验。
- 增加接口测试、服务测试和数据库迁移脚本验证。
- 如果后续继续做大，再考虑 Redis、对象存储、搜索引擎等增强组件。

## 13. 总结

本项目已经形成一个可本地运行、功能较完整、结构较清晰的小型毕业设计系统。它的核心价值不在于追求企业级复杂度，而在于：

- 以 Spring Boot 单体架构完整实现商城、社区、攻略和后台；
- 在小规模样例数据下形成注册、登录、浏览、购买、入库、积分、发帖、评论、推荐、签到、打赏等完整业务链；
- 具备较好的论文撰写基础和答辩展示价值；
- 参数、结构和数据规模均符合毕业设计项目应有的实现深度与工作量。
