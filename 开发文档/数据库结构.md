# 数据库结构

本文件以当前代码中的实体映射和现阶段实际使用场景为准，替代根目录原有的零散数据库说明、修复指南和临时 SQL 注释。

## 1. 数据库概览

- 数据库名：`bishe`
- 字符集：`utf8mb4`
- 推荐存储引擎：InnoDB
- 当前开发配置：JPA `ddl-auto=update`
- 推荐维护方式：以 `开发文档/SQL脚本` 下的新脚本为准，避免继续混用历史脚本

## 2. 当前主表

| 表名 | 作用 | 关键字段 | 备注 |
| --- | --- | --- | --- |
| `users` | 用户主表 | `username`、`email`、`password`、`points`、`role`、`status` | 当前已切换为加密密码，兼容旧明文自动升级 |
| `user_sessions` | 登录会话表 | `user_id`、`session_token`、`expires_at` | 保存 Cookie 对应的会话 |
| `cart_items` | 购物车表 | `user_id`、`game_id`、`unit_price`、`selected` | 当前按数字商品单份购买处理 |
| `orders` | 订单表 | `order_no`、`user_id`、`payable_amount`、`points_earned`、`status` | 当前支付方式为模拟支付 / 免费领取 |
| `order_items` | 订单项表 | `order_id`、`game_id`、`game_name`、`subtotal` | 下单时固化商品快照 |
| `user_games` | 用户游戏库表 | `user_id`、`game_id`、`order_id`、`acquired_at` | 支撑数字商品入库 |
| `point_transactions` | 积分流水表 | `user_id`、`change_amount`、`balance_after`、`type` | 记录积分变动 |
| `categories` | 游戏分类表 | `name`、`parent_id`、`sort_order`、`status` | 支持父子分类 |
| `games` | 游戏主表 | `name`、`category_id`、`price`、`discount_price`、`tags` | `category_id` 仍保留兼容意义 |
| `game_categories` | 游戏-分类关联表 | `game_id`、`category_id` | 多分类扩展使用 |
| `banners` | 轮播图表 | `title`、`image_url`、`link_url`、`type` | 首页轮播管理使用 |
| `posts` | 讨论广场帖子表 | `user_id`、`game_id`、`title`、`status` | 对应 `/api/posts` |
| `comments` | 讨论广场评论表 | `post_id`、`user_id`、`parent_id`、`status` | 支持楼中楼回复 |
| `community_sections` | 社区板块表 | `name`、`icon`、`sort_order`、`status` | 对应社区首页板块卡片 |
| `community_posts` | 社区板块帖子表 | `section_id`、`user_id`、`title`、`status` | 对应 `/api/community/posts` |
| `community_comments` | 社区板块评论表 | `post_id`、`user_id`、`parent_id`、`status` | 社区板块评论体系 |

## 3. 关系说明

### 用户相关

- `users 1 -> n user_sessions`
- `users 1 -> n cart_items`
- `users 1 -> n orders`
- `users 1 -> n user_games`
- `users 1 -> n point_transactions`
- `users 1 -> n posts`
- `users 1 -> n comments`
- `users 1 -> n community_posts`
- `users 1 -> n community_comments`

### 商城相关

- `games 1 -> n cart_items`
- `orders 1 -> n order_items`
- `games 1 -> n order_items`
- `orders 1 -> n user_games`
- `categories 1 -> n games` 通过 `games.category_id` 保持兼容
- `games n -> n categories` 通过 `game_categories` 支持多分类
- `banners` 独立存在，通过 `link_url` 指向页面或资源

### 讨论与社区相关

- `posts 1 -> n comments`
- `comments 1 -> n comments` 通过 `parent_id` 实现回复
- `community_sections 1 -> n community_posts`
- `community_posts 1 -> n community_comments`
- `community_comments 1 -> n community_comments` 通过 `parent_id` 实现回复

## 4. 数据约定

### 游戏标签与系统要求

- `games.tags` 统一按逗号分隔字符串保存
- `games.system_requirements` 统一按文本保存
- 不再推荐使用历史 JSON 字段方案

### 评分

- 当前统一按 5 分制维护
- 历史数据里如果出现大于 5 的分值，应执行修复脚本归一化

### 商城交易与积分

- 当前商城流程采用“加入购物车 -> 模拟支付 -> 自动入库 -> 累积积分”的方案
- `orders.status` 目前主要使用 `PAID`
- `orders.payment_method` 目前主要使用 `MOCK` 和 `FREE`
- `users.points` 为当前积分余额，`point_transactions` 为积分变动明细

### 分类

- 历史上存在重复分类、旧分类状态为空、多分类关系不同步的问题
- 现在应优先维护 `game_categories`
- `games.category_id` 可视为兼容字段，不应再作为唯一事实来源

## 6. 历史脚本中的预留表说明

下列表在旧版 `database_init.sql`、`sample_data.sql` 或其他临时 SQL 中出现过，但当前主线代码并未直接依赖，因此没有继续放入新的“主初始化脚本”里：

- `products`
- `admin_logs`
- `community_likes`
- `community_favorites`

处理原则如下：

- 如果后续商城闭环要继续做，这些表建议在新需求稳定后单独重建设计，而不是继续依赖旧脚本拼补。
- 如果仅用于毕业设计展示，当前主表已经足够支撑首页、详情、登录、购物车、订单、游戏库、积分、社区、后台管理等主线演示。

## 7. 推荐维护原则

1. 以本目录的新 SQL 为准，不再直接执行根目录历史脚本。
2. 新增字段优先同步实体类与初始化脚本，再考虑 JPA 自动更新。
3. 涉及删除或合并数据的操作，先做查询确认，再单独编写迁移脚本，不要混入基础初始化脚本。
